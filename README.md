# E2E 3D Notes

Feed-forward / end-to-end multi-view 3Dに関する研究上の疑問，実験計画，結果，失敗，訂正を継続的に蓄積するための共有リポジトリである．

特定の手法や一つのプロジェクトを説明する場所ではない．各テーマはGitHub Issueを入口とし，コードや論文，会話履歴を知らない読者でも，背景から実験の判定基準まで理解できる形で保存する．

## 記録を見る

- [GitHub Issues](https://github.com/KohsukeIde/e2e-3d-notes/issues): テーマごとの背景，仮説，実験内容，現在の結果を読むための索引
- [`reports/`](reports/): Issueだけでは収まらない詳細な実験レポート
- [`issues/`](issues/): GitHub Issue本文の保存版
- [`figures/`](figures/): レポートとIssueで使用する図
- [`data/`](data/): 図や集計の入力となるCSV，JSON，出所情報
- [`scripts/`](scripts/): 集計と図を再生成するスクリプト
- [実験レポートの保存ルール](docs/reporting_contract.md): 書き方，集計単位，再現性，訂正方法の詳細

## このリポジトリに残すもの

- 既存研究の結果から生じた，具体的な研究上の疑問
- その疑問を説明する仮説と，仮説同士を区別する実験
- 入力，出力，比較対象，固定条件，データ，評価指標を含む実験設定
- 成功結果だけでなく，否定的な結果，失敗した実行，予想と異なった挙動
- 後から誤りが分かった主張と，訂正理由
- 何が分かり，何がまだ分からないかという境界

## 新しいIssueを追加する

[New issue](https://github.com/KohsukeIde/e2e-3d-notes/issues/new)から，一つの研究上の問いにつき一つのIssueを作る．タイトルは，読者が内容を予測できる具体的な疑問文にする．

Issue本文は，原則として次の順序で書く．

1. **状態**: 仮説のみ，実験計画，実行中，結果あり，訂正済みのいずれかを明記する．
2. **背景**: 既存手法が何を入力し，何を出力し，どこに未解決の疑問があるかを書く．
3. **仮説**: 何が結果を決めていると考えるのかを書く．
4. **自明でない点**: 単なる既知手法の言い換えではないか，どの別説明が残るかを書く．
5. **比較方法**: 何を固定し，何だけを変えるかを書く．
6. **具体的な設定**: モデル，データ，画像枚数，解像度，camera convention，評価指標，seedなどを書く．未定の値は未定と記す．
7. **判定基準**: どの結果なら仮説を支持し，棄却し，保留するかを実験前に書く．
8. **結果**: 集計単位，失敗数，図表，予想外だった点を含める．未実施なら結果がないことを明記する．
9. **言えないこと**: 一つのモデルやデータセットから一般化できない範囲を書く．
10. **一次資料**: 原論文，公式コード，使用データへのリンクを付ける．

長い導出，完全な設定表，多数の図が必要な場合は`reports/`へ詳細レポートを置き，Issueからリンクする．図や数値を追加する場合は，可能な範囲で入力データと生成スクリプトも保存する．

## 記述上の原則

- 本文は自然な「だ・である」調で書き，句読点は「，．」へ統一する．
- 分野で標準的な用語は英語のままでよいが，日本語で自然に書ける箇所へ不要な英単語を挟まない．
- 内部の変数名，実験名，実行ジョブ名，commit hashを，説明なしに本文へ出さない．
- 論文に書かれた事実，手元で再現した結果，新しい比較実験，結果からの解釈を混同しない．
- 結果が変わった場合は以前の主張を黙って削除せず，何が誤りで，現在何が残るかを記録する．

## 根拠の区分

各記録では，必要に応じて次の区分を使う．本文中へ繰り返し挿入せず，冒頭または末尾の根拠一覧にまとめる．

| 区分 | 意味 |
|---|---|
| `paper` | 原論文または公式コードに明記された事実 |
| `local-reproduction` | 公開モデルを再実行して得た結果 |
| `local-experiment` | 条件を管理した手元の比較から得た結果 |
| `inference` | 観測結果から導いた解釈．追加検証が必要 |
| `retracted` | 後の監査で無効になった主張．理由とともに履歴を残す |

## リポジトリへファイルを追加する場合

1. 先にGitHub Issueを作り，研究上の問いと実験内容を記録する．
2. 詳細レポート，データ，図，スクリプトを必要な範囲だけ追加する．
3. Issueから追加ファイルへリンクし，読者がIssueを入口に全体を追えるようにする．
4. 実験後は結果と判定を同じIssueへ追記する．別の問いが生じた場合は，新しいIssueへ分ける．

学習済みモデル，データセット，上流コードは再配布しない．各データのライセンスは配布元に従う．

## 図と保存データの検査

```bash
python3 -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt
make check
```

`make check`は，保存データ，本文で参照する集計値，生成した図の整合性を検査する．モデルの学習や推論から完全に再実行できることを意味するものではない．
